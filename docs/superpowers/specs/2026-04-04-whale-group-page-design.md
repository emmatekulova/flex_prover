# Join Whale Group Page — Design Spec

**Date:** 2026-04-04
**Status:** Approved

## Problem

Traders who have generated a verified proof on Flex Prover want to use that proof as a "golden ticket" to access private trading communities (Discord, Telegram). Currently there is no way to gate access based on a verified on-chain proof.

## Solution

A new "Join Whale Group" tab in the existing Next.js frontend. The user connects their wallet, pastes a link to their on-chain proof, and the frontend verifies two things:

1. The wallet address embedded in the proof matches the currently connected wallet
2. The attested metric in the proof meets the threshold (e.g. PnL > 20%)

If both checks pass, the invite link to the private group is revealed. No new smart contracts. No backend.

## Assumptions

- The on-chain proof (generated by the TEE) contains both:
  - The user's wallet address
  - An attested metric (PnL % or similar)
- The exact proof format will be specified separately. This page is designed to be adapted once that format is confirmed.
- The invite link(s) are static and hardcoded in the frontend for now (Discord and/or Telegram).

## User Flow

```
Connect Wallet
     ↓
Paste proof link (tx hash or block explorer URL)
     ↓
Frontend fetches proof from Coston2 chain
     ↓
Extract: wallet address + attested metric
     ↓
Check 1: wallet address == connected wallet?  → fail: "Proof doesn't belong to this wallet"
Check 2: metric >= threshold?                 → fail: "Metric below required threshold"
     ↓
Both pass → Reveal invite link
```

## UI Layout

New tab next to the existing Verify tab. The page has three states:

**Idle**
- Wallet connect button (if not connected)
- Text input: "Paste your proof link"
- Submit button: "Verify Access"

**Verifying**
- Loading indicator while fetching proof from chain

**Result**
- Success: green badge + Discord/Telegram invite link button
- Failure: red badge + specific error message (wallet mismatch or threshold not met)

## Components

| Component | Action |
|---|---|
| Wallet connect | Reuse existing Reown AppKit setup |
| SIWX wallet ownership | Reuse `lib/siwx-verify.ts` — confirms the connected wallet owns the address |
| Proof fetching + decoding | Reuse `components/verifier.tsx` logic — fetch tx, decode attestation event |
| `WhalePage` component | New — orchestrates the flow above |
| Proof link input | New — simple text input, accepts tx hash or full block explorer URL |
| Access result display | New — shows success (invite link) or failure (error message) |

## Verification Logic

```
1. Parse proof link → extract tx hash
2. Fetch tx from Coston2 RPC
3. Decode AttestationPublished event → extract walletAddress + metric
4. Check: walletAddress === connectedWallet (from SIWX session)
5. Check: metric >= THRESHOLD (e.g. 0.20 for 20% PnL)
6. If both pass: reveal INVITE_LINK
```

The threshold and invite link are constants in the component, easy to update.

## What Is Not In Scope

- Minting SBTs or any on-chain action from this page
- Building a Discord/Telegram bot
- A backend or database
- Preventing invite link sharing (out of scope — the TEE proof is the trust anchor, not the invite link itself)
- Supporting multiple proof types simultaneously (one type at a time, format TBD)
